# /pipeline — Portfolio View

Read all pipeline directories and render a status dashboard. No writes —
this is a read-only snapshot of the current portfolio.

---

## Usage

```
/pipeline
```

### HTML Board

The portfolio is also available as an HTML kanban board at `pipeline/index.html`,
auto-generated by a PostToolUse hook whenever pipeline files change. Open it
with `open pipeline/index.html` to see the visual board.

The generator script lives at `scripts/generate-pipeline-board.js`.

---

## Process

### Phase 1 — Scan

1. Glob `pipeline/*/` to find all idea directories
2. For each directory, detect stage by checking which files exist:

| Files present | Stage |
|--------------|-------|
| `learn.md` only | Learned |
| `learn.md` + `explore.md` + `sketches/` | Explored |
| `learn.md` + `explore.md` + `assess.md` | Assessed |
| `prove.md` | Proven |
| `ship/` | Shipped |

3. Check for `.state.json` — if present, a skill is in progress:
   read `skill` and current phase to show "In Progress: /explore (prototype)"

### Phase 2 — Extract

For each idea, read available files:

**From `opportunity.md`** (if exists):
- JTBD: the blockquote subtitle
- Blocking questions: count items in "Decide before building" section
- Last updated: footer timestamp

**From `decision-log.md`** (if exists):
- Total decisions: count Q/A entries

**From `pipeline/{slug}/eval/`** (if exists):
- Latest eval score: read most recent date-stamped scorecard (`YYYY-MM-DD.md`), extract "Overall: X.X / 5.0"
- Eval date
- Latest per-phase health: tail `phase-checks.jsonl` for the most recent passed/failed counts

### Phase 3 — Render

Present the portfolio table:

```markdown
# Pipeline Portfolio

| Idea | Stage | JTBD | Blocking | Decisions | Score | Updated |
|------|-------|------|----------|-----------|-------|---------|
| [{slug}](pipeline/{slug}/opportunity.md) | {stage} | {jtbd truncated to 60 chars} | {N} | {N} | {X.X} | {date} |

**Summary:** {N} ideas — {N} Assessed, {N} Explored, {N} Learned
```

**If an idea has no opportunity.md**, link to the directory instead and
pull the JTBD from `learn.md` (the JTBD field).

**If a skill is in progress** (`.state.json` exists), show the stage as:
`Explored (in progress: prototype)` — the base stage plus the active phase.

### Phase 4 — Highlights

After the table, add:

```markdown
## Needs Attention

- **{slug}**: {N} blocking questions unanswered — run `/prove` or resolve manually
- **{slug}**: Last updated {N} days ago — stale?
- **{slug}**: Eval score {X.X} — below 3.0, review quality

## Next Actions

- {slug}: Ready for `/N` — [recommend next skill based on stage]
```

Only show "Needs Attention" if there are items. Skip empty sections.

---

## Principles

1. **Read-only.** This skill never writes files. It's a lens, not a tool.
2. **Scan fast.** Read only what's needed — headers, footers, counts. Don't
   parse full file contents unless extracting a specific field.
3. **Link everything.** Every idea name is a clickable link to its opportunity.md
   or directory. The PM should be able to jump straight to any idea.

# /compare — Prioritize Pipeline Ideas

Compare all ideas in the pipeline side-by-side using their Scoring sections.
Produce a ranked prioritization with rationale.


## Usage

```
/compare
/prioritize
/compare water-subscription referral-rewards guided-onboarding
```

Without arguments: compares all ideas in the pipeline.
With arguments: compares only the named ideas.


## Process

### 1. Scan Pipeline

Read all `pipeline/*/opportunity.md` files (or only specified slugs).

Extract from each:

| Field | Source |
|-------|--------|
| Title | `# Build:` heading |
| JTBD | Blockquote subtitle |
| **Conviction** | `.briefdata.json` `conviction.score` (0-100) — the primary ranking signal |
| **Verdict** | `.briefdata.json` `conviction.verdict` (Pursue / Needs / Kill) |
| **Biggest gap** | `.briefdata.json` `conviction.gap.dim` |
| Impact | Scoring section (secondary lens) |
| Complexity | Scoring section (secondary lens) |
| Stage | Which files exist (learn.md → Learned, explore.md → Explored, assess.md → Assessed, prove.md → Proven, ship/ → Shipped) |
| Opportunity size | Key numbers from learn.md |
| Feasibility | Overall verdict from prove.md (if exists) |
| Open questions | Count from Decide before building table |

If an idea is missing the Scoring section, flag it: "[idea] has no Scoring —
run the latest pipeline skill to add it, or I'll exclude from ranking."

### 2. Score & Rank

**Primary signal: conviction.** Read `conviction.score` (0-100) and
`conviction.verdict` from each idea's `.briefdata.json`. Conviction is computed
by one rigor-gated engine — it already blends Problem, Size, Feasibility, and
Evidence quality, and it cannot run high unless claims are sourced. It IS the
bet quality, so it is the ranking spine (not a hand-set Impact x Confidence formula).

- **Rank by `conviction.score`, descending.**
- **Exclude `kill` verdicts** from the active ranking — list them separately under "Parked / killed". A clean Kill is a result, not a failure.
- Use the opportunity.md **Impact / Complexity** only as a SECONDARY sequencing lens among ideas of *similar conviction*: prefer lower Complexity (quicker wins) and higher Impact to break ties and decide what to do first.
- If an idea has no `conviction` yet (pre-/3), note it as "unscored — needs /assess" and rank it below scored ideas.

Present ranked table (conviction-first):

```markdown
| Rank | Idea | Conviction | Verdict | Impact | Complexity | Stage | Biggest gap |
|------|------|-----------|---------|--------|------------|-------|-------------|
| 1 | [title] | 87 | Pursue | H | L | Assessed | Size |
| 2 | [title] | 67 | Needs | H | M | Proven | Evidence |
| 3 | [title] | 52 | Needs | M | M | Assessed | Evidence |
```

### 3. Context Cards

For each idea, show a compact card:

```markdown
### [Rank]. [Title] — Score: [X]

> [JTBD]

**Stage**: [stage] | **Feasibility**: [verdict or "not assessed"]
**Opportunity**: [size from learn.md]
**Open questions**: [count]
**Scoring rationale**:
- Impact [H/M/L]: [one-line from Scoring section]
- Confidence [H/M/L]: [one-line]
- Complexity [H/M/L]: [one-line]
```

### 4. PM Adjustment

Ask: "This ranks by conviction, with Impact/Complexity as a sequencing tiebreaker. Want to:"

Options:
1. **Sequence by quick wins** — among similar-conviction ideas, sort by lowest Complexity first
2. **Override a rank** — PM sees something the conviction model missed (log why)
3. **Add constraints** — e.g., "must ship before Q3", "needs < 2 teams"
4. **Filter by stage or verdict** — e.g., only Pursue, or only ideas past /assess
5. **Keep as-is**

If PM adjusts: re-rank and re-present. Never silently override conviction — if the PM wants to rank a low-conviction idea highly, capture the reason (it's a real signal the model may be missing).

### 5. Recommendations

After PM confirms ranking:

```markdown
## Recommendations

**Start next**: [top-ranked idea] — [one-line on why it's the best bet]

**Needs work before starting**: [ideas with Yellow/Red feasibility or many open questions]
- [idea]: [what's blocking — specific open question or dependency]

**Consider cutting**: [lowest-ranked ideas with weak rationale]
- [idea]: [why — e.g., low confidence with no path to validate]

**Stage gaps**: [ideas that haven't run enough pipeline stages for a fair comparison]
- [idea]: only at [stage] — run /[next skill] to get a better score
```

### 6. Output

Save to `pipeline/prioritization.md`:

```markdown
# Pipeline Prioritization — {date}

## Ranking

| Rank | Idea | Score | Stage | Key bet |
|------|------|-------|-------|---------|
| 1 | [title] | [score] | [stage] | [one-line from JTBD] |

## Scoring Details

| Idea | Impact | Confidence | Complexity | Score | Feasibility |
|------|--------|------------|------------|-------|-------------|
| [title] | [H/M/L] — [rationale] | ... | ... | [score] | [verdict] |

## Formula

`{formula used}` {with any weight adjustments noted}

## Constraints Applied

{Any PM-specified constraints, or "None"}

## Recommendations

{From step 5}

*Generated by /compare, {date}*
*Ideas compared: {count}*
*Formula: {formula}*
```

Present: "Here's the prioritization — [top idea] leads with a score of [X].
See `pipeline/prioritization.md` for the full breakdown."


## Principles

1. **Data from Scoring sections.** Don't re-assess — use the scores each
   skill already computed. The pipeline did the thinking; this skill does
   the comparison.
2. **Transparent formula.** Show the math. Let PM challenge and adjust.
   No black boxes.
3. **Stage-aware.** Earlier-stage ideas have less-informed scores — flag
   this explicitly. A Learned idea's confidence is less reliable than
   an Explored idea's.
4. **No opinion without evidence.** If Scoring is missing, flag but don't
   guess. Encourage running the relevant skill.
5. **Prioritization is a conversation.** The first ranking is a starting
   point. The PM brings context the formula can't capture — constraints,
   politics, timing, strategic bets.

# /cost — Pipeline Cost Report

Show execution statistics across the pipeline — skill runs, agent invocations,
MCP calls, and connector failures. Helps understand pipeline usage patterns
and identify flaky connectors.


## Usage

```
/cost                    # full report across all ideas
/cost water-subscription # report for a single idea
```


## Process

### Full Report (no slug)

1. Run `node scripts/pipeline-cost-report.js`
2. Display the output to the PM
3. If connector failure rate > 5%, recommend running `/setup --health`

### Per-Idea Report (with slug)

1. Run `node scripts/pipeline-cost-report.js {slug}`
2. Display the output to the PM
3. If the idea has failures, summarize which connectors failed and how often

## Data Sources

1. **`pipeline/{slug}/execution-log.md`** — Phase-by-phase execution data written by every pipeline skill. Contains agent invocations, MCP calls, and connector failures per phase.
2. **`pipeline/{slug}/opportunity.md`** — The Pipeline cost section (summary table maintained by `_shared/protocols/pipeline-cost.md`).

The script parses execution logs using the format defined in the CLAUDE.md Execution Log section. Each `## /N skill — date` header starts a skill run, and each `### Phase N: name` contains the per-phase metrics.


## Principles

1. **Execution data, not token counts.** We track phases, agents, and MCP calls — not raw LLM tokens. These are more actionable.
2. **Failure patterns matter.** Connector failures indicate infrastructure issues, not PM mistakes.
3. **Lightweight.** This is a read-only report — no state changes, no artifacts written.

# /replay — Decision Replay

Walk through the decision history of a pipeline idea as a structured narrative.
Useful for onboarding new PMs, explaining a bet to stakeholders, or reviewing
your own thinking after a break.


## Usage

```
/replay water-subscription               # full replay
/replay water-subscription --stage /explore    # only /explore decisions
/replay water-subscription --highlights  # key decision points only
```


## Process

### Step 1 — Load

1. Verify `pipeline/{slug}/` exists
2. Read `decision-log.md`
3. Read `opportunity.md` for context (JTBD, scoring, current stage)
4. If `--stage` specified, filter to entries from that skill only

### Step 2 — Parse & Structure

Parse `decision-log.md` into structured entries:

```
Entry {
  type: "question" | "data_pull" | "cascade" | "runtime_failure"
  skill: "/learn" | "/explore" | etc.
  question: string (for Q&A entries)
  answer: string (for Q&A entries)
  context: string (for data pulls)
  findings: string (for data pulls)
  timestamp_hint: string (from When field)
}
```

Group entries by skill stage.

### Step 3 — Identify Key Decisions

Mark entries as "key decisions" if they match any of:
- JTBD selection (mentions "job", "occasion", "outcome" in Q or A)
- Direction choice (mentions "direction", "variation", "chose", "selected")
- Kill criteria (mentions "kill", "threshold", "stop if")
- Sizing/evidence (data pull entries with quantified findings)
- Risk acceptance (mentions "risk", "accept", "proceed despite")
- Scope decisions (mentions "build", "don't build", "later", "cut")

### Step 4 — Present Walkthrough

If `--highlights`:
  Show only key decisions, one per message, with:
  ```
  Key Decision ({skill}, Q{N})

  Q: {question}
  A: {answer}

  Why this matters: {1-sentence explanation of impact on the idea}
  ```

If full replay:
  Walk through chronologically, grouping by skill stage:
  ```
  ## /1 learn — Problem Definition

  {N} decisions made during grounding.

  **Q1**: {question}
  -> {answer}

  **Q2**: {question}
  -> {answer}

  Data: {data pull summary}

  ---

  ## /2 Explore — Solution Exploration

  ...
  ```

### Step 5 — Summary

End with a decision summary:
```
Decision Summary for {slug}:
- Stage: {current stage}
- Total decisions: {count}
- Key decision points: {count}
- Data pulls: {count}
- Skills completed: /learn, /explore, /assess
- Open questions: {count from decisions.md if exists}
```


## Principles

1. **Narrative, not dump.** Don't just list Q&A — tell the story of how the idea evolved.
2. **Evidence highlighted.** Data pulls and their findings get special treatment.
3. **Key decisions surfaced.** The PM shouldn't have to read 50 entries to find the 5 that matter.
4. **Stage boundaries clear.** Show where each skill's contribution starts and ends.
